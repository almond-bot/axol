---
title: "teleop"
description: "Launch a VR teleoperation session on the robot or simulator."
---

Launches a VR teleoperation session. When started, the hostname (`.local`) and local IP address are printed — enter either of these in the VR app at [axol.almond.bot](https://axol.almond.bot) to connect.

Runs on the **real robot by default**; pass `--sim` for the browser visualizer. The full robot config is exposed via draccus, so every gain, stiffness, and gripper limit is reachable from the CLI or a config file — see [Command configuration](/cli/configuration).

<Tip>
  For a guided, step-by-step walkthrough including the VR controller layout, see the [Teleoperation quickstart](/quickstart/teleop).
</Tip>

<Warning>
  **Before opening the VR app**, accept the self-signed HTTPS certificate in the VR browser by navigating to `https://<hostname>.local:8000` or `https://<local-ip>:8000` and proceeding past the security warning.
</Warning>

<Note>
  **Network tip:** If VR tracking feels jittery or packets arrive in bursts, configure the following on your router/access point:
  - **DTIM interval** → `1`
  - **Beacon interval** → `100` ms
  - **WMM APSD (U-APSD)** → disabled

  These settings prevent the AP from buffering packets between beacon intervals, which causes intermittent delivery delays that are especially noticeable for latency-sensitive VR traffic.
</Note>

| Flag | Description |
|---|---|
| `--sim` | Use the software visualizer instead of the real robot. Bare `--sim` works (or `--sim true`). Omit to run on real hardware (the default). |
| `--left_channel <ch\|null>` | CAN channel for the left arm; `null` disables it. Default: `CAN_LEFT`. |
| `--right_channel <ch\|null>` | CAN channel for the right arm; `null` disables it. Default: `CAN_RIGHT`. |
| `--axol.left_stiffness S` | Compliance↔stiffness blend for the left arm in `[0, 1]`. Scalar (`0.8`) or a 7-element list (`[0.8,0.8,0.5,0.5,0.2,0.2,0.0]`, one per arm joint in `Joint` enum order). `0` = fully compliant; `1` = pre-tuning industrial gains; `0.5` (default) is the geometric mean. See [`AxolConfig.left_stiffness`](/api/robot). |
| `--axol.right_stiffness S` | Same, for the right arm. |
| `--axol.left.gripper.torque_limit FLOAT` | Max torque (Nm) for the left gripper in POSITION_FORCE mode (default: 0.5). Use `right` for the right gripper. |
| `--axol.<side>.<joint>.<gain> VALUE` | Override any per-joint gain, e.g. `--axol.left.elbow.kp 60`. `<side>` is `left`/`right`; `<joint>` is `shoulder_1…wrist_3`; `<gain>` is `kp`/`kd`/etc. |
| `--teleop.position_multiplier FLOAT` | Scale factor on hand **position** (not orientation): the end-effector target moves this many times as far as your hand. `1.0` (default) is 1:1; `2.0` moves the arm twice as far, helping cover the robot's full reach when the arm is longer than the operator's. |
| `--teleop.rotation_multiplier FLOAT` | Scale factor on hand **orientation** (not position): the end-effector rotates this many times as far as your wrist. `1.0` (default) is 1:1; `2.0` rotates the arm twice as far. |
| `--cameras DICT` | Map camera slots to the serial numbers of locally attached ZED cameras, e.g. `--cameras "{overhead: 41234567, left_arm: 41234568}"` (any of `overhead` / `left_arm` / `right_arm`). When set, teleop opens those cameras and relays them to the headset over WebRTC (overhead + wrist views, switched with the right thumbstick). A stereo ZED X overhead is detected automatically from its serial and relayed as `overhead_left` / `overhead_right` for per-lens rendering. Requires `pyzed` ([`zed.install`](/cli/zed-install)) and the `video` extra. Default: `{}` (no headset video). |
| `--resolution {SVGA,HD1080,HD1200}` | Capture resolution for all cameras. Default `null`: each camera's SDK default (HD1200 on GMSL). |
| `--log_level {DEBUG,INFO,WARNING,ERROR}` | Default: `INFO` |
| `--config_path PATH` | Load a whole-config JSON/YAML file; CLI overrides layer on top. See [Command configuration](/cli/configuration). |

```bash
axol teleop                                       # real robot
axol teleop --sim                                 # browser visualizer
axol teleop --axol.left_stiffness 0.8
axol teleop --axol.left.elbow.kp 60 --axol.right.gripper.torque_limit 0.7
axol teleop --teleop.position_multiplier 2.0      # scale hand motion 2x
axol teleop --teleop.rotation_multiplier 2.0      # scale hand rotation 2x
axol teleop --left_channel null                   # disable the left arm
axol teleop --cameras "{overhead: 41234567, left_arm: 41234568, right_arm: 41234569}"
axol teleop --cameras "{overhead: 41234567}"      # stereo overhead auto-detected, per-lens
axol teleop --config_path my_teleop.json          # whole-config file
```

## Camera streaming to the headset

With `--cameras` set, teleop opens the locally attached ZED cameras by serial number and relays them to the headset over WebRTC, so the operator sees the robot's cameras instead of (or alongside) passthrough. The right thumbstick switches views — see the [VR Interface guide](/guides/vr-interface#camera-views) for the view system and controls. Requires `pyzed` ([`zed.install`](/cli/zed-install)) and the `video` extra (`uv sync --extra video`).

On a Jetson with the [zed-gstreamer](https://github.com/stereolabs/zed-gstreamer) plugins installed (the hosted installer builds them automatically), mono cameras stream through a **GPU-resident pipeline**: `zedxonesrc` grabs frames straight into GPU memory, NVENC encodes them there, and Python only ever forwards the already-encoded H.264 — about 5 ms from grab to encoded frame at HD1200@60, versus ~26 ms for the SDK relay. Raw pixels never cross into Python, so this path cannot share a camera with the SDK (which is why `collect-data` keeps the SDK path).

The whole relay — camera pipelines plus WebRTC sending — runs in a **dedicated subprocess** (`almond_axol.vr.video_proc`), the same isolation pattern used for the IK solver: sending video pushes thousands of RTP packets per second through Python, and doing that inside the teleop process measurably halves the IK rate. Only SDP signaling crosses the process boundary, so the control loops are unaffected by streaming.

A **stereo overhead** (auto-detected from the overhead serial) joins the same GPU-resident relay through Stereolabs' `zedsrc`: it grabs both eyes side-by-side in one zero-copy NV12 buffer, splits them on the VIC, and encodes each eye with its own NVENC session — exposed as `overhead_left` / `overhead_right` for per-lens rendering. This keeps the stereo head out of the teleop process too; only if the stereo `zedsrc` plugin is missing does the session fall back to the in-process SDK path (which, because the relay and SDK can't share the WebRTC manager, pulls the wrist cameras back in-process as well).

When the plugins are missing (or for a stereo overhead), the relay falls back to the **frame-driven SDK path**: each camera frame is encoded and sent the moment it's captured (teleop also asks the cameras for 60 fps capture, falling back to the camera default when unsupported), and the headset is asked to render with minimal jitter buffering. On a Jetson this still encodes with the **hardware H.264 encoder** (NVENC via GStreamer's `nvv4l2h264enc`) instead of aiortc's software codec. Either way, teleop pins the NVENC and VIC engine clocks to their maximum (the default Jetson governor scales them for throughput, not latency, which roughly triples per-frame encode time); this needs root — automatic under the hosted `axol serve` install; when running `teleop` or `collect-data` from a terminal with cameras configured, you may be prompted for `sudo` once at startup. If escalation isn't possible, a warning log prints the manual command. If no GStreamer hardware encoder is installed at all, the relay falls back to software encoding — `SVGA` is then the recommended resolution.

</Note>
