Secure WebSocket server (WSS) that receives `VRFrame` JSON messages from the VR app. A self-signed TLS certificate is auto-generated in `~/.almond/vr/certs/` on first use. This module can be used standalone to read raw VR data without full teleoperation â€” useful for custom control loops or data collection.

```python
from almond_axol.vr import VRServer, VRServerConfig
```

```python
import asyncio
from almond_axol.vr import VRServer, VRServerConfig

async def main():
    async with VRServer(VRServerConfig(port=8000)) as vr:
        while True:
            frame = vr.get_frame()
            if frame is not None:
                print(frame.l_ee, frame.r_ee)
            await asyncio.sleep(0.01)

asyncio.run(main())
```

Or use a callback instead of polling:

```python
def on_frame(frame):
    print(frame.l_grip, frame.r_grip)

async with VRServer() as vr:
    vr.set_on_frame(on_frame)
    await asyncio.sleep(float("inf"))
```

## `VRFrame` fields

| Field | Type | Description |
|---|---|---|
| `l_ee` / `r_ee` | `VRPose` | 6-DOF end-effector pose (position + quaternion) |
| `l_elbow` / `r_elbow` | `VRPosition` | 3D elbow positions |
| `l_grip` / `r_grip` | `float [0, 1]` | Gripper commands |
| `l_lock` / `r_lock` | `bool` | Grip toggles (engage/disengage tracking) |
| `reset` | `bool` | Rising edge triggers a reset move |
| `state` | `VRState` | `TELEOP`, `DATA_COLLECTION`, or `RECORDING` (headset-driven); `SAVING` and `ERROR` are server-pushed only |

## Server â†’ headset feedback

The server can push a state override to all connected headsets at any time using `VRServer.broadcast_text()`. The headset interprets messages of the form `{"type": "state", "value": "saving"}` as a state override that blocks recording controls. `{"type": "state", "value": "error"}` shows an error indicator in the headset UI. `{"type": "state", "value": "data_collection"}` re-enables controls after saving. The `AxolVRTeleop.send_feedback_state(state)` helper wraps this for all `VRState` values.

Teleop additionally broadcasts `{"type": "tracking", "value": true|false}` whenever the engage toggle changes, so the headset knows when the robot is being controlled â€” it uses this to only allow repositioning the camera screens (a trigger gesture) while tracking is disengaged.

## Camera video to the headset

The server can stream camera frames to the headset over **WebRTC**. Register per-camera frame sources with `set_video_sources()` â€” each source is a connected camera (or stereo eye) exposing fixed `width` / `height` / `fps` plus `wait_next(after_ts, timeout_ms)`, which blocks until a frame newer than `after_ts` is captured (the `FrameSource` protocol, mirroring the subset of `ZedCamera` the relay uses). A `ZedCamera` satisfies it directly:

```python
vr.set_video_sources({
    "overhead":  overhead_cam,
    "left_arm":  left_cam,
    "right_arm": right_cam,
})
```

The existing `/ws` channel is reused for SDP signaling (no extra ports): the headset sends `webrtc-request`, the server replies with a `webrtc-offer` carrying one video track per source (labelled by camera name), and the headset answers with `webrtc-answer`. A stereo overhead is registered as two sources, `overhead_left` / `overhead_right`, which the VR app renders per-lens.

Pass `None` or `{}` to turn video off. [`axol teleop --cameras`](/cli/teleop) and [`collect-data`](/cli/collect-data) wire this up automatically.

### Pipeline

Both teleop and data collection grab ZED frames with the **Python SDK** (`ZedCamera` â†’ RGB numpy) and hand them to a single GStreamer-native manager (`almond_axol.vr.gst_webrtc.GstWebRTCManager`). For each connected headset it builds one self-contained GStreamer pipeline â€” an `appsrc` per camera, `nvvidconv` â†’ `nvv4l2h264enc` (the Jetson **NVENC** hardware encoder), and one `webrtcbin`. The relay pulls each source's `wait_next` so a frame is encoded the instant it's captured (no fixed-rate sampling staleness), then **WebRTC is sent entirely from GStreamer** â€” RTP/SRTP/ICE all run in C inside `webrtcbin`, so **no encoded frames ever return to Python** and the control loops never pay for video traffic. ICE is non-trickle (host candidates only) so gathering completes in milliseconds on a LAN; the gathered candidates are spliced into the offer SDP before it's sent.

The whole relay runs in a **dedicated subprocess** (`almond_axol.vr.video_proc.VideoRelayProcess`, registered via `set_video_manager()`): even though sending happens in C, isolating the camera grab/encode pump keeps the teleop control loops at full rate. Only SDP signaling crosses the process boundary.

Requires the GStreamer WebRTC stack â€” PyGObject, `webrtcbin` (`gstreamer1.0-plugins-bad`), libnice (`gstreamer1.0-nice`), and the Jetson `nvv4l2h264enc` element â€” installed once by [`axol gst.install`](/cli/gst-install) (the host installer runs it). If it's unavailable, `set_video_sources()` logs a warning and leaves video disabled. The NVENC/VIC engine clocks must also be pinned to their maximum for low-latency encode ([`axol jetson.setup`](/cli/jetson-setup), run at boot by the installer's systemd unit); the default Jetson devfreq governor scales them for throughput, which otherwise roughly triples per-frame encode latency.

## `VRServerConfig` fields

| Field | Default | Description |
|---|---|---|
| `port` | `8000` | WSS listen port |
| `certfile` | auto | Path to TLS certificate; `None` uses the auto-generated cert |
| `keyfile` | auto | Path to TLS private key |

<Warning>
  Before opening the VR app, accept the self-signed certificate by navigating to `https://<hostname>.local:8000` in the VR browser and proceeding past the security warning.
</Warning>
