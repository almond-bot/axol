# Almond Axol v0.1.2

Changes landed since the v0.1.1 release, focused on remote teleoperation, dataset replay, and camera/streaming flexibility.

## New

- **`replay-dataset` command** — Replays a recorded LeRobot episode on the robot: pick a dataset and episode index and it returns to rest, streams the episode's recorded actions onto the arms at the dataset's fps, then parks at rest. It's the inverse of `collect-data` and needs no cameras. Pass `--loop` (a toggle in the control panel) to replay continuously — returning to rest between takes — until stopped with `Ctrl+C` or the UI's **Stop**, then return to rest. Available from the web control panel under **Operate**, between Collect Data and Run Policy.
- **Public remote teleoperation** — Operate the robot over the internet, not just the local network. Pose, control, and camera video stream to a remote operator over WebRTC, with the headset's control data channel re-negotiating itself after a backoff (and falling back to a WebSocket pose path) when the peer connection drops. Reconnecting resets the pose interpolator and clears stale frames so a new session doesn't blend buffered poses into the IK.
- **Cartesian-space observation/action toggle** — `observe_cartesian` makes each arm's state and action a 6-axis end-effector pose (position + axis-angle rotation, FLU) plus gripper, instead of 7 joint angles. Observations run FK on the joint cache; Cartesian actions are resolved to joint targets via IK in `send_action`. Available for `collect-data`, `run-policy`, and `replay-dataset` (which auto-detects a Cartesian dataset and replays it through IK). The IK solver is built lazily and warmed up ahead of the control loop so the first action doesn't stall.
- **Stereo wrist support** — Adds stereo wrist cameras and stereo data collection.
- **Full camera configuration in the control panel** — The Cameras dialog now owns all camera config as a single source of truth: separate streaming (capture → headset) and recording (dataset) resolutions, plus independent per-camera Stream and Record controls (on/off for a mono camera, per-eye L/R for a stereo one). An operator can stream a camera without recording it (or vice versa), stream both eyes for depth while recording only one, pick which cameras feed the headset vs. the dataset, or turn either branch off entirely. The headset shows only the views actually being streamed, with a global "Connecting cameras" indicator.
- **WiFi fallback for Quest-over-USB pose streaming** — The headset streams every pose frame over both the wired USB tunnel and the network socket at once, tagged with a shared monotonic sequence number. The VR server processes each frame once, from whichever link delivers it first — USB while it's live, WiFi the instant USB goes quiet — so a mid-session cable disconnect fails over to WiFi instantly with no reconnect, and `Enter VR` is no longer gated on the cable.

## Improvements

- **Constant-quality dataset video encoding** — On-Jetson recordings were several times larger than the old upper-computer ones because the NVENC encoder ran at a fixed bitrate budget regardless of scene motion. Switched `nvv4l2h264enc` to constant quality (fixed QP 30) to mirror LeRobot's libx264 `crf=30`, so the bitrate tracks scene complexity and a mostly-static teleop scene encodes small again.
- **Faster, reliable Stop** — `/api/op/stop` now returns immediately (the session goes to "stopping") and a background watchdog SIGKILLs the op's child subprocesses (video relay, dataset recorder, IK worker) after a short grace, so a blocked teardown actually unwinds instead of hanging the request for up to 20s and leaving the op running. The watchdog snapshots its kill targets under a lock so it can't kill a newly started op's children, and the web panel shows a disabled "Stopping…" button and badge while polling until the op is truly gone.
- **Durable ports** — Tracks the sockets it binds and reclaims a leaked listening port on the next bind, so a stopped or timed-out op no longer wedges the next run with "address already in use" — including the VR server's WebSocket port, which runs on an in-process thread the force-kill can't reach.
- **Better ZED handling** — More robust ZED camera enumeration and bring-up.
- **Damiao motor read retries** — `_read_register` now re-sends and waits afresh for up to 5 attempts, so a single missed reply during an `enable` burst (when several motors read registers at once) no longer aborts the whole enable.
- **Quieter installer** — `gst.install` no longer emits a scary "GStreamer ZED camera stack is still unavailable" warning for `zedxonesrc`, which the very next provision step installs; it now verifies only its own deliverables and reports the source elements as an informational note. The Jetson CPU-governor warning reports how many cores failed and points at the pending-MAXN / reboot path before the manual override.

## Internal

- **Package reorganization** — Modules moved to homes that match what they are, with no logic changes: the recorder subprocess to `recording/`, camera/streaming infra to `video/`, domain constants (`Joint` enum, CAN names, URDF paths) to `constants.py`, and hardware bring-up scripts from `test/` to `diagnostics/`.
